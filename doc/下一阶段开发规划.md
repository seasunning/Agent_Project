# 下一阶段开发规划

## 1. 文档目的

本文档用于结合当前系统已经落地的功能，规划接下来一段时间的开发重点、阶段目标、任务拆解和实施顺序，帮助项目从“已跑通主流程”逐步推进到“更稳定、更可用、更接近产品化”。

---

## 2. 当前项目所处阶段判断

根据当前代码与已有文档，系统已经具备以下能力：

- Phase 2：需求智能分析
- Phase 3：设计方案自动生成
- Phase 4：代码原型自动生成
- Phase 5：工程落盘与压缩包下载

这说明项目已经不是单纯的页面演示或接口骨架，而是已经具备一条从输入需求到输出工程原型的基本链路。

因此，下一阶段开发重点不应继续无序扩功能，而应转向：

> **先补齐主链路可用性，再增强生成质量与产品化能力。**

---

## 3. 下一阶段总体目标

下一阶段建议聚焦以下三个核心目标：

### 目标 1：把主流程做成真正可闭环

让用户可以稳定完成如下流程：

1. 输入原始需求
2. 生成结构化需求结果
3. 生成设计方案
4. 选择技术栈
5. 生成代码原型
6. 落盘为本地工程
7. 下载压缩包
8. 根据启动说明运行生成项目

### 目标 2：提升代码生成结果的可信度

不仅“生成出来”，还要能让用户知道生成结果是否完整、是否基本可用、问题大概在哪里。

### 目标 3：补齐工作区与会话管理能力

让用户能够保存、恢复、继续编辑已有结果，而不是每次刷新都从头开始。

---

## 4. 开发优先级规划

建议按 `P0 -> P1 -> P2` 分层推进。

---

## 5. P0：优先完成主链路闭环

P0 是当前最重要的一层，建议优先完成。

### 5.1 生成项目可用性验收

#### 目标

验证当前系统生成出的项目是否真的具备基本可运行性，而不仅仅是“生成了文件”。

#### 具体任务

- 从 `backend/generated_projects/` 中选择 2~3 个代表性项目作为样本
- 检查生成项目目录结构是否完整
- 检查关键文件是否存在
- 检查 `README.md`、`STARTUP_STEPS.txt` 是否能指导启动
- 尝试安装依赖并启动项目
- 记录失败原因和共性问题

#### 建议产出

新增文档：

- `doc/生成项目验收记录.md`

#### 价值

这一步可以帮助后续明确：

- 代码生成问题主要发生在哪一类项目
- 是提示词问题、文件规划问题，还是落盘补全问题
- 哪些技术栈组合更容易生成成功

---

### 5.2 增加生成结果校验机制

#### 目标

在代码生成完成后，系统自动对结果做一层基础检查。

#### 具体任务

后端建议增加一个“生成结果校验”步骤，至少包括：

- 文件数量统计
- 必要文件存在性检查
- 启动说明是否存在
- 常见配置文件是否存在
- 不同语言场景下的简单规则校验

例如：

- Python 项目检查是否存在 `requirements.txt`
- 前端项目检查是否存在 `package.json`
- FastAPI 项目检查是否存在入口文件
- Vue/React 项目检查是否存在前端入口

#### 前端展示建议

在代码生成结果页增加“校验结果摘要”，例如：

- 生成成功，基础结构完整
- 生成成功，但缺少部分配置文件
- 已落盘，但启动前需人工补全依赖

#### 价值

让系统从“只负责生成”升级为“生成 + 初步判断可用性”。

---

### 5.3 优化失败提示与异常反馈

#### 目标

让用户在失败时知道是哪里出了问题，而不是只看到泛化报错。

#### 具体任务

梳理并细分以下失败场景：

- 模型调用失败
- 网络连接失败
- 模型输出 JSON 解析失败
- 文件生成失败
- 落盘失败
- 压缩包不存在或下载失败

#### 展示建议

前后端错误提示中增加：

- 当前失败步骤
- 失败原因摘要
- 建议用户下一步操作

例如：

- “模型已返回，但结构化解析失败，建议查看原始输出后重试”
- “工程已生成，但压缩包打包失败，请先使用目录结果”
- “后端未连接成功，请先确认服务和环境变量配置”

#### 价值

降低调试成本，提高系统可诊断性。

---

## 6. P1：提升代码生成可用性

P1 建议在 P0 完成后尽快推进，因为这部分最能提升系统实际价值。

### 6.1 支持单文件重生成

#### 目标

当某个文件生成效果不理想时，允许用户只重生成当前文件，而不是整个项目重来。

#### 具体任务

- 在代码生成页支持对单个文件发起“重新生成”
- 将当前设计方案、技术选型、项目规划和目标文件路径一起传给后端
- 后端新增单文件生成接口
- 生成完成后替换该文件内容并保留其余结果

#### 价值

这个功能对实际使用体验提升很大，能显著减少重复生成成本。

---

### 6.2 增强文件生成上下文一致性

#### 目标

提升前后端文件之间的一致性，减少命名、接口、模块定义不统一的问题。

#### 具体任务

- 在生成单文件时显式传入完整文件规划结果
- 在提示词中强调项目名称、模块命名、接口命名统一
- 增加关键实体、关键接口的共享上下文
- 对生成文件的路径和语言进行统一标准化

#### 价值

提高整体工程质量，降低“每个文件都像单独生成的”割裂感。

---

### 6.3 增加生成后基础检查

#### 目标

在落盘后进行简单的工程级检查，而不是只做结构性检查。

#### 具体任务

可根据技术栈逐步增加：

- Python 文件基础语法检查
- `requirements.txt` 是否可解析
- `package.json` 是否为合法 JSON
- 核心入口文件是否可读取
- 文件编码/空内容检查

#### 价值

进一步提升生成结果质量与可维护性。

---

## 7. P1：补齐工作区与会话能力

### 7.1 增加会话保存能力

#### 目标

让用户可以保存一整次工作流结果，而不是只依赖页面内状态。

#### 具体任务

- 为一次工作流引入“项目会话”概念
- 保存以下内容：
  - 原始需求文本
  - 需求分析结果
  - 设计方案结果
  - 技术选型结果
  - 代码生成结果
  - 落盘结果
- 支持从历史会话恢复

#### 存储方案建议

初期可先落地为：

- 前端本地存储增强版
- 或后端简单 JSON 文件存储

后续再逐步升级到数据库。

---

### 7.2 增加历史记录页

#### 目标

让系统具备最基本的项目管理能力。

#### 具体任务

- 增加历史记录/会话列表页
- 展示每次生成的项目名称、时间、阶段状态
- 支持重新打开某次记录
- 支持删除不需要的历史记录

#### 价值

这会让系统更接近“工作台产品”，而不只是“单次生成工具”。

---

## 8. P2：增强需求与设计阶段体验

### 8.1 需求分析页增强

#### 建议任务

- 支持结构化字段直接编辑
- 支持待确认问题逐条确认/标记已处理
- 支持基于当前结果再次分析某一段需求
- 支持版本对比（原始结果 vs 编辑结果）

### 8.2 设计方案页增强

#### 建议任务

- API 定义改为更结构化的表格编辑方式
- 数据实体支持字段级操作
- Mermaid 错误提示更具体
- 模块关系展示更直观
- 支持导出更完整的设计文档

### 8.3 工作台首页增强

#### 建议任务

- 展示最近一次会话
- 展示最近一次落盘项目
- 展示后端、模型、生成状态概览
- 展示各阶段完成度

---

## 9. 中长期方向规划

在主链路稳定之后，建议从以下两个方向中选择一个作为下一步主线。

### 方向 A：真正做成多智能体编排系统

如果项目名称强调“Multi Agents”，后续可以补齐：

- 需求分析 Agent
- 架构设计 Agent
- 技术选型 Agent
- 代码生成 Agent
- 质量审查 Agent
- 工作流编排器

这会让系统从“多阶段生成工具”升级为“多智能体开发协同平台”。

### 方向 B：优先把代码生成质量做强

如果更偏重实用价值，可优先投入：

- 单文件重生成
- 错误自动修复
- 配置文件自动补全
- 代码检查与生成后修复
- 更稳定的工程模板生成

这会让系统更快具备“生成即可用原型”的产品价值。

---

## 10. 建议的近期实施顺序

建议按如下顺序推进：

### 第一阶段：稳定主链路

1. 生成项目验收
2. 增加结果校验
3. 优化错误提示

### 第二阶段：提升生成能力

4. 支持单文件重生成
5. 增强文件一致性
6. 增加基础语法/配置检查

### 第三阶段：增强产品化体验

7. 增加会话保存
8. 增加历史记录页
9. 优化需求页与设计页编辑体验

### 第四阶段：明确中长期路线

10. 决定走“多智能体编排”还是“高质量代码生成平台”方向

---

## 13. 规划结论

当前系统最合理的下一步不是继续无节制扩功能，而是优先把“需求 -> 设计 -> 代码 -> 落盘 -> 验证”这条主链路做稳做实。

可以用一句话概括下一阶段的开发策略：

> **先让系统生成结果更稳定、更可判断、更可继续使用，再进入真正的多智能体编排或更高级代码生成能力建设。**

如果执行顺序正确，项目将会从“已有原型能力”逐步升级为“可持续迭代的软件开发辅助平台”。
